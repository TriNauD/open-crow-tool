# QA 手测 → E2E 冒烟（规范化流程）

## 和现有规则怎么接

- **手动/半自动**验收文档仍写在 `dev/active/需求名称/需求名称-qa.md`（见 `product-delivery-pipeline` 技能）。
- **单元/契约测**继续在仓库根目录 `__tests__/**/*.test.ts`（Vitest），管「函数、API 头、纯逻辑」。
- **浏览器里点出来的流程**（打开站、点按钮、跳页）用本技能管：建议 **Playwright**（比传统 Selenium 省不少皮实活，和 Next 社区常见搭配一致）。

三者分工：**Vitest = 快、不拉浏览器**；**E2E = 慢、但像真人**；**手测 = 扩展/邮件/极难自动化**那几类。

---

## 推荐目录（落地代码时）

```
e2e/                    # 或 tests/e2e/，团队定一个即可
  smoke.spec.ts          # 冒烟：全站健康 + 最小路径
  fixtures/              # 可选：共用登录态、mock
playwright.config.ts     # 根目录，与 Next 平级常见
```

未装 Playwright 前，本技能只当**流程与命名约定**；真正加依赖、改 `package.json` 时按 `dependency-and-lockfile.mdc` 单独说明。

---

## 和 QA 文档的「对表」规矩（核心）

1. 在 **`需求名称-qa.md` 的 §0.2 手测最小路径**里，给可自动化的步骤加稳定编号：  
   `E2E-01`、`E2E-02`…（或沿用你已有的 `H-01` 再在括号写 `E2E`）。
2. 在 E2E 源文件里，**每个用例 / 或 `test.describe` 上注释**：对应 `E2E-xx` + 文档章节。  
   这样以后改产品，**要么改文档，要么改脚本**，两边能对着查。
3. **不能自动化的**（Chrome 未打包扩展、收真实邮件、扫码等）仍在 §0.2 写明 **「仅手测，原因：…」**，不要硬写进冒烟脚本里凑数。

`§0.1 自动化` 里加两行即可：

| 类型 | 命令 | 与 QA 关系 |
|------|------|------------|
| 单元/契约 | `npm run test` | 对应 §0.1 与 `__tests__` |
| 浏览器冒烟 | `npx playwright test`（落地后） | 对应 §0.2 中带 `E2E-xx` 的步骤 |

## 第一阶段「冒烟」建议范围（可先做这些）

在 **`BASE_URL`** 指向本机、Preview 或 Staging 的前提下，**尽量不接真实发信、不加载未打包扩展**：

| 编号 | 做什么（大白话） | 为什么算冒烟 |
|------|------------------|--------------|
| E2E-01 | 打开首页，能出标题/主 CTA，HTTP 不报错 | 站活着 |
| E2E-02 | 打开 `/login`、`/register`，有表单或预期占位 | 鉴权相关页在 |
| E2E-03 | 未登录用户访问 `/notebook` 行为与产品一致（能进或重定向，按你们 PRD） | 主功能入口在 |
| E2E-04 | 可选：GET `/api/health` 若以后有加；或只测页面不直接打私网 API | 看你们是否愿暴露轻量健康检查 |

**先明确不做（避免第一期就翻船）：**

- 不自动跑 **已打包 Chrome 扩展** 的全链（要另开阶段，用 Playwright 带 extension path）。
- 不自动**收真实邮箱点链接**（要接测邮箱 API 或 test-only token）。
- 不在仓库里**写死**测试邮箱 A、B 的**密码/验证码**；一律 **环境变量 / GitHub Actions Secrets**。

你提供的「测试邮箱 A、B」适合放在：本地 `.env.test.local`（勿提交）+ CI **Secrets**。

## 和 CI 的关系

- 现状：`npm run verify` 已管 **lint + 单元测 + 双包 build**。
- E2E 上 CI 要额外：装浏览器、跑 Playwright，**更慢、更容易 flaky**。常见做法：  
  - **PR 上先不跑** E2E，只跑 `verify`；或  
  - **nightly** / **人工点「跑 E2E」**；或  
  - 只在 **`dev` 合并** 或 **发版前** 跑。  
  定一条团队规矩即可，写进本仓库的 `playwright.config.ts` 注释或 `docs/tech` 里一小段。
