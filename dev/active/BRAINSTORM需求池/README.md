## 收件箱 → 分拣（进池前）

**为什么要多这一步**：聊天、脑暴、路上想到的点**如果直接当正式需求**，会拖死排期；**如果只记微信**，会忘。所以先经**收件箱**，再进下面的「需求池 / Issue / 立项」。

### 三条流（别混成一个筐）

| 类型 | 先放哪 | 定型后去哪 |
|------|--------|------------|
| **想法 / 可能的大需求** | 个人可记 [`notes/需求与反馈收件箱.md`](../../../notes/需求与反馈收件箱.md)，或补本池下的 [`待办-*.md`](./待办-B-笔记分类.md) / 上方总表加一行 | **排期敲定要做了** → 按下文「从需求池立项」建 `dev/active/<需求简称>/` |
| **明确的 bug**（能复现或有清晰现象） | 优先 **[GitHub Issues](https://github.com/TriNauD/open-crow-tool/issues)**，打 `bug` | 修：小走 `fea/*` 或 `hotfix/*`；不必先写进待办长篇 |
| **小优化 / 体验** | Issue 一条，或本池表格/待办中加一行短语 | 和 B/C/D 条目一样排优先级 |

约定：**大块想法默认不要口头承诺上线日期**；**能复现的 bug 不要用「待办 md 的长文」顶替 Issue**。

**`notes/brainstorm/`**（如 `BRAINSTORM-todo.md`）为历史或临时脑暴草稿；**排期、立项与谁做谁后，以本目录总表 + [`待办-*.md`](./待办-B-笔记分类.md) 为准**。分拣时把仍有效的点写进池子或上面「想法」流，过时则删或标注已处理，避免与本池**双轨并行**。

### 节奏（轻量）

1. **当场（约 1 分钟）**：记一行到收件箱笔记，或开 Issue。
2. **每周 10～15 分钟（或站会）**：收件箱分拣 → **进池 / 建 Issue / 丢弃**（丢弃可写一句原因，免得明年再问）。
3. **决定开发时**：必须通过「从需求池立项」或 Issue 认领，走 `dev-workflow`（含单独立项目录）。

**`dev/active/` 语义**：该目录下可有**多种已立项、进行中的需求**（例如 `日报/`），并不全是 BRAINSTORM。**本池**只承担 B/C/D **待排期条目**的索引、`待办-*.md` stub 与路线图；不要误以为 `dev/active` 整块都是「需求池本体」。

## 从需求池立项（必做）

自本池取用某一条目准备开发（含排期敲定、立项后）时，**必须为该项单独建档**，不得在 `待办-*.md` 里顶替 plan/context/tasks，也不得与池中其他未选条目共用同一套 active 文档。

**流程简述**

1. 在条目尚未正式开工前，占位与简述可保留在对应的 [`待办-*.md`](./待办-B-笔记分类.md)（或本条表中的占位链接）。
2. **一旦立项、进入开发与验收轨道**：在 **`dev/active/<需求简称>/`** 建立**独立目录**（需求简称示例：`笔记本分类`、`划词上下文`，与 `.cursor/rules/git-branching.mdc` 的分支前缀可对应）。
3. **目录内必有**（与 `.cursor/rules/dev-workflow.mdc` 阶段 3 一致）：
   - `<需求简称>-plan.md`
   - `<需求简称>-context.md`
   - `<需求简称>-tasks.md`
4. 进入 QA：**`<需求简称>-qa.md`**；结项：**迁 `dev/done/`**、补 `dev/logs/…-log.md`、按需更新 `docs/PRD.md` / `docs/PLAN.md` / 相关分卷（见该项目既有产品交付管道）。
5. **回填本页**：下方「文档位置」列中，将该条从「仅待办 stub」**更新为** `dev/active/<需求简称>/` 主链（可一并保留待办文件作背景索引）。

**阶段 A 说明**：A-1 / A-2 已随「BRAINSTORM 阶段 A」整包归档在 [`dev/done/BRAINSTORM阶段A/`](../../done/BRAINSTORM阶段A/)，不拆成两个独立 `dev/active/*` 目录；**自阶段 B 起**，表中每条新开工均按上表**单独立项**建档。
